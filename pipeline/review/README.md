# Lovänya — visual-pipeline exploration (UI-PC, CPU, 2026-07-06)

Scratch findings from validating the **Style Me wardrobe-thumbnail** flow on the
UI-PC (no capable GPU — 940MX/2GB). Everything here is generated by the scripts
in `../scripts/`; re-run them to regenerate. These are **exploration artifacts,
not production** — the real pipeline runs on the 4060 Ti (seconds/photo).

## What's here
- `index.html`, `review.jsonl`, `pdf/` — **Claude-gold vs Qwen-8B judgment
  review** (`build_review.py`, `build_review_pdf.py`). 121 outfits; 7 have a real
  8B verdict; the rest are Claude-only. For pressure-testing judgments and
  collecting **human ratings** → `data/ratings.jsonl` (the taste the set lacks).
- `extract/` — **CPU multi-garment extraction** (`extract_garments.py`): one
  outfit photo → GroundingDINO detects each garment → rembg cutout + colour tags.
- `lookcards/`, `extract/lookcards.html` — **standalone HTML of the LookCard
  "style card"** (`build_lookcards_html.py`; a faithful port of
  `lovanya/lib/lookcard.ts` + `LookCard.tsx`).
- `birefnet/` — **rembg vs BiRefNet matting bake-off** (`simulate_birefnet.py`).

## Findings
1. **Multi-garment extraction works in shape.** One photo → N categorised
   garments (e.g. `sample_r1c5` → 5: top/bottom/shoes/bag/accessory;
   `b4_r4c4` → 3). This validates the Style Me auto-fill flow end to end.
2. **Per-garment cutout quality is the weak link.** rembg (u2net) over-erases
   busy garment crops → faint, patchy cutouts (see `extract/`).
3. **The BiRefNet bake-off was run on WHOLE photos, and was inconclusive.**
   Per-garment crops need the slow CPU detection step, so the fair test wasn't
   run. On whole photos the two models are comparable by coverage (opaque-pixel
   %: within a few points, except `b4_r4c4` where BiRefNet kept ~50% more). So
   **whole-photo matting is NOT the bottleneck — the per-garment crop is.**
4. **Conclusion:** the definitive fix is **box-prompted SAM2 on the 4060 Ti**
   (mask the *exact* GroundingDINO box), not swapping the whole-photo
   salient-object model. Re-testing **BiRefNet on tight garment crops** on the
   GPU is worth doing; a whole-photo model swap alone won't move it.

## Design decisions carried in from this session
- `idealized is fine` for the thumbnail (opens generative options later), but
  **graceful highlight/spotlight fallback** is locked: never show a broken
  cutout — degrade to the outlined real crop.
- DeepSeek is not an image generator (LLM/VLM); if generative is ever used, the
  real contenders are self-hosted Flux/SDXL inpaint or a paid product-shot API —
  weighed against the free, faithful SAM2/BiRefNet baseline first.

## Honest limits
- CPU-only timings: GroundingDINO ~12 min/photo, BiRefNet ~1 min/photo, rembg
  ~1–2 s/photo. Not production speed. These stand-ins validate the *shape* and
  surface the weak link; they do not set the quality ceiling.
- `review/` holds ~7.7 MB of generated artifacts (regenerable via the scripts).
